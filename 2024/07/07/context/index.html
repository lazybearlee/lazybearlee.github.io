Go Context 不完全解析

发表于2024-07-07|更新于2024-07-07|GoNote
|阅读量:

context标准库

设计目的

总的目的就是根据goroutine的调用树，并在树中传递通知和元数据。

  • 退出通知机制——通知可以传递给整个调用树上每个goroutine
  • 传递数据——数据可以与传递给整个goroutine调用树上每个goroutine

基本数据结构

通过逐层封装对象构建出一个表示整个调用树的对象，这个对象可以在每个节点都被访问到.

context中实现了canceler接口的具体类型是可以被取消的，而 *cancelCtx 和 *timeCtx 就实现了这个接口。

valueCtx

分析一下构建流程——以WithCancel为例

当我们调用语句 ctx, cancel := context.WithCancel(context.Background()) 时， 我们首先传入一个 contex.Background() 参数。这个参数代表的函数会返回一个新的 Context 。这个 Context 实质上是一个 emptyCtx 。它所实现的所有方法都返回零值。

而在传入这个 backgroundCtx 之后，会做这样一步：构造具有取消方法的Ctx，并封装这个取消方法。

而在构造Ctx的函数中，需要首先判断parent是否为空（也就是说context不允许用户从空的context创建子context）。然后会新建一个 cancelCtx ，用于在后面设置前向传播关系。

在 propagateCancel 主要干了以下几件事情：

  • 设置子context的Context为父context
  • 调用父context的Done方法，Done返回一个通道，该通道在代表此上下文完成的工作应该被取消时关闭。如果这个上下文永远不能被取消，Done可能会返回nil。对Done的连续调用返回相同的值。Done通道的关闭可以在cancel函数返回后异步发生。
  • 一旦在创建的时候parent就已经完成工作被取消，那么child就会被立即取消。
    • 然后我们调用 parentCancelCtx 返回父级的底层 *cancelCtx。它通过查找 parent.Value(&cancelCtxKey) 来找到最内层的 *cancelCtx，然后检查 parent.Done() 是否匹配该 *cancelCtx。 （如果不匹配，那么 *cancelCtx 已经被包装在一个自定义实现中，提供了一个不同的 done 通道，在这种情况下，我们不应该绕过它。）而如果找到了这个 *cancelCtx ，就意味着已经有一个专门的协程负责处理 Done 信道并关闭子context（并且没有被自定义过），接下来就应该给 p 上锁了，避免这时候其他例程取消 p 的执行或更改状态。同时，也需要再次判断 p 是否已经被取消，没被取消我们才需要向 p 的 children 写入新的child，然后可以直接返回。
    • 如果上面一步没有匹配成功，那么就要看 parent 有没有实现 AfterFunc 方法（用类型断言）。如果实现了，恭喜，可以包装一下了，我们可以给 parent 包装一个 stop 方法，用于在某个事件之后再去取消 context ，然后同样可以直接返回。
    • 最后，新开一个协程用于处理 parent 发过来的消息取消context（还有自己的，不过只接收不处理）。

通过这样一个流程，我们就可以逐步构建出一个调用树。
